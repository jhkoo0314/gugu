# FEATURE_SPEC_V2.md

# gugu 추가 기능 명세서
버전: v0.2 ~ v0.3 초안  
대상: 기획자, 프론트엔드 개발자, AI 코딩 에이전트  
기준: 현재 v0.1 구현 완료 상태

---

## 1. 문서 목적

이 문서는 현재 완료된 구구단 연습 웹앱(gugu)에 아래 추가 기능을 구현하기 위한 기능 요구사항과 상태/로직 확장 기준을 정의한다.

대상 기능:
- 직접 입력 모드
- 타이머 모드
- 연속 정답 보상
- 효과음 on/off
- 결과 화면 시각 강화
- 학습 기록 저장

이 문서의 목표는 다음과 같다.

1. 기존 v0.1 로직을 깨지 않고 기능을 확장할 수 있어야 한다.
2. 상태 추가 시 책임이 명확해야 한다.
3. 모바일 우선 UX를 유지해야 한다.
4. 구현 우선순위와 범위를 나눠 개발이 쉬워야 한다.

---

## 2. 버전 권장 범위

기능을 한 번에 다 넣기보다 아래처럼 나누는 것을 권장한다.

### v0.2
- 직접 입력 모드
- 타이머 모드
- 연속 정답 보상
- 효과음 on/off
- 결과 화면 시각 강화

### v0.3
- 학습 기록 저장

이렇게 나누는 이유:
- v0.2는 세션 내 인터랙션 중심 기능
- v0.3는 저장 구조와 기록 조회가 포함되는 기능

---

## 3. 공통 확장 원칙

추가 기능 구현 시 아래 원칙을 유지한다.

1. 핵심 상태는 여전히 `src/app/page.tsx`에 둔다.
2. 기존 `start / quiz / result` 화면 구조는 유지한다.
3. 한 문제는 한 번만 채점하는 규칙은 절대 깨지지 않아야 한다.
4. 새 기능은 “세션 규칙”과 “표시 규칙”을 분리해서 구현한다.
5. 기능을 끄거나 켤 수 있는 옵션은 가능한 시작 전에 결정한다.
6. 아이가 이해하기 어려운 설정 UI는 늘리지 않는다.

# 4. 직접 입력 모드

## 4.1 목적
객관식 외에 숫자를 직접 입력해서 구구단을 풀 수 있도록 한다.

## 4.2 사용자 가치
- 암기 수준을 더 정확히 확인할 수 있다.
- 객관식 힌트 없이 순수하게 답을 떠올리는 연습이 가능하다.
- 익숙해진 아이에게 한 단계 높은 연습 모드를 제공할 수 있다.

## 4.3 기능 요구사항

시작 화면 또는 모드 선택 영역에서 풀이 방식을 선택할 수 있어야 한다.

지원 모드:
- 객관식 모드
- 직접 입력 모드

직접 입력 모드에서는:
- 보기 4개 대신 숫자 입력 UI를 보여준다.
- 답을 입력하고 확인 버튼을 눌러 채점한다.
- 모바일에서도 입력하기 쉬운 숫자 입력 구조를 사용한다.

## 4.4 권장 UX
초기 구현은 일반 키보드 입력보다 아래 둘 중 하나를 권장한다.

### 권장안 A
`input type="number"` + 확인 버튼

장점:
- 구현이 빠름
- 로직 단순

### 권장안 B
숫자 키패드형 버튼 UI

장점:
- 초등학생 친화적
- 모바일에서 더 편함

초기 구현은 **A안**이 적절하다.

## 4.5 상태 추가

```ts
type AnswerMode = 'multiple-choice' | 'input';
````

추가 상태:

* `answerMode: AnswerMode`
* `inputAnswer: string`

## 4.6 동작 규칙

### 시작 전

* 사용자가 풀이 방식을 선택한다.

### 문제 화면

* 객관식 모드면 기존 방식 유지
* 직접 입력 모드면 숫자 입력창과 확인 버튼 노출

### 채점

* 입력값을 숫자로 변환하여 정답 비교
* 빈 값이면 채점 불가
* 이미 채점된 문제면 중복 채점 불가

## 4.7 예외 처리

* 빈 입력값 상태에서 확인 버튼 클릭 시 무시
* 숫자가 아닌 값은 허용하지 않음
* 음수는 허용하지 않음
* 입력 완료 후 채점되면 입력창 비활성화

## 4.8 UI 요구사항

* 입력창은 크고 단순해야 한다.
* 숫자는 크게 보여야 한다.
* 확인 버튼은 분명하게 보여야 한다.
* 직접 입력 모드에서도 현재 문제, 진행률, 피드백 흐름은 동일하게 유지한다.

## 4.9 구현 포인트

* 기존 `handleAnswerSelect`를 재사용할 수 있도록 입력값도 결국 `number`로 넘기게 설계한다.
* `inputAnswer`는 다음 문제 이동 시 초기화한다.

# 5. 타이머 모드

## 5.1 목적

문제를 제한 시간 안에 풀도록 하여 집중력과 반응 속도를 높인다.

## 5.2 사용자 가치

* 긴장감이 생겨 몰입도가 올라간다.
* 구구단 숙련도를 빠르게 점검할 수 있다.
* 놀이형 느낌이 강해진다.

## 5.3 기능 요구사항

타이머 모드는 선택형 옵션으로 제공한다.

지원 방식 예시:

* 일반 모드
* 타이머 모드

또는:

* 문제당 제한 시간 설정
* 세션 전체 제한 시간 설정

초기 구현은 **문제당 제한 시간 방식**을 권장한다.

## 5.4 권장 설정

* 문제당 10초 또는 15초
* 시간이 다 되면 자동 오답 처리 후 다음 문제로 넘어가거나, 시간 초과 상태를 보여준 뒤 다음 문제 버튼 활성화

초기 구현은 아래 방식 권장:

* 시간이 0이 되면 자동 오답 처리
* “시간이 끝났어요!” 메시지 표시
* 자동으로 다음 문제로 넘어가지 않고, 다음 문제 버튼을 누르게 함

이 방식이 더 안정적이다.

## 5.5 상태 추가

추가 상태:

* `isTimerMode: boolean`
* `timeLeft: number`
* `timePerQuestion: number`

## 5.6 로직 규칙

### 문제 시작 시

* 타이머 모드가 켜져 있으면 `timeLeft = timePerQuestion`

### 카운트다운

* 1초마다 감소
* `isAnswered === true`가 되면 카운트다운 중지

### 시간 초과 시

* 아직 답하지 않았다면 자동 오답 처리
* `selectedAnswer = null` 유지 가능
* 오답 기록에 `userAnswer`는 특수값 처리 필요

권장:

```ts
userAnswer: -1
```

또는 타입 확장:

```ts
userAnswer: number | null
```

더 자연스러운 방식은 `number | null` 이다.

## 5.7 타입 변경 권장

```ts
type WrongAnswer = {
  questionId: string;
  multiplicand: number;
  multiplier: number;
  correctAnswer: number;
  userAnswer: number | null;
  reason?: 'wrong-answer' | 'timeout';
};
```

## 5.8 UI 요구사항

* 상단에 남은 시간 표시
* 시간 바(progress style) 또는 숫자 표시
* 시간 부족 시 시각적으로 조금 더 강조
* 빨간 경고 느낌보다 부드러운 핑크/피치 계열 경고 권장

## 5.9 예외 처리

* 채점 후에도 타이머가 계속 돌면 안 됨
* 다음 문제 이동 시 타이머 초기화
* 결과 화면 이동 시 interval 정리
* 페이지 전환 시 timer cleanup 필수

## 5.10 구현 포인트

* `useEffect`로 `timeLeft` 감소 처리
* `screen`, `currentIndex`, `isAnswered`, `isTimerMode` 변화에 맞춰 정리(cleanup) 필요

# 6. 연속 정답 보상

## 6.1 목적

연속으로 맞힐수록 성취감을 느끼게 하고, 계속 풀고 싶게 만든다.

## 6.2 사용자 가치

* 반복 학습이 덜 지루해진다.
* 실시간 동기부여가 생긴다.
* 아이가 “잘하고 있다”는 느낌을 받게 된다.

## 6.3 기능 요구사항

* 연속 정답 횟수를 계산한다.
* 특정 횟수마다 칭찬 또는 보상 효과를 보여준다.

예시:

* 2연속: “좋아요!”
* 3연속: “대단해요!”
* 5연속: “완전 최고예요!”

## 6.4 상태 추가

* `streakCount: number`
* `bestStreak: number`

## 6.5 로직 규칙

* 정답이면 `streakCount += 1`
* 오답이면 `streakCount = 0`
* `bestStreak`는 최대값 유지
* 시간 초과도 streak 끊김으로 처리

## 6.6 표시 규칙

* 정답 직후 피드백 문구 아래 보조 문구 노출 가능
* 특정 연속 횟수에 배지/별/작은 애니메이션 표시 가능
* 결과 화면에서 최고 연속 정답 수를 보여줄 수 있음

## 6.7 UI 요구사항

* 너무 과한 게임 UI는 피함
* 작은 별, 반짝이, 배지 정도만 허용
* 피드백 문구 옆 또는 상단에 짧게 표시

## 6.8 예시 문구

* 2연속 정답! 좋아요!
* 3연속 정답! 멋져요!
* 5연속 정답! 정말 대단해요!

## 6.9 구현 포인트

* 채점 로직에서 정답/오답 분기 시 함께 처리
* 결과 화면에 `bestStreak` 표시 가능

# 7. 효과음 on/off

## 7.1 목적

효과음이 있는 즐거움은 유지하되, 사용자가 원할 때 끌 수 있게 한다.

## 7.2 사용자 가치

* 공공장소나 조용한 환경에서도 사용 가능
* 보호자가 소리를 제어할 수 있음
* 사용자 선호에 맞게 조절 가능

## 7.3 기능 요구사항

* 효과음 사용 여부를 켜고 끌 수 있어야 한다.
* 정답, 오답, 연속 정답 보상 등에 소리 적용 가능
* 효과음이 꺼져 있으면 어떤 상황에서도 재생되지 않아야 한다.

## 7.4 상태 추가

* `soundEnabled: boolean`

## 7.5 기본값

권장 기본값:

* `true`

단, 사용성에 따라 첫 진입은 `false`로 두는 것도 가능하다.
현재 이미 효과음을 구현했다면 v0.2에서는 **기본값 true**가 자연스럽다.

## 7.6 UI 위치

효과음 토글은 아래 중 하나에 둔다.

권장 위치:

* 시작 화면 우측 상단 또는 하단 작은 토글
* 퀴즈 화면 상단 작은 아이콘 버튼

권장 방향:

* 시작 화면과 퀴즈 화면 둘 다 토글 가능

## 7.7 로직 규칙

* `soundEnabled === true`일 때만 재생 함수 실행
* 정답, 오답, 보상음 재생 전에 항상 상태 확인

## 7.8 구현 포인트

* 사운드 재생 함수를 공통 유틸로 감싸는 것이 좋다.
* 예:

```ts
playCorrectSound(soundEnabled)
playWrongSound(soundEnabled)
```

또는:

```ts
if (soundEnabled) audio.play()
```

## 7.9 확장 권장

향후 localStorage 저장 시 효과음 설정도 함께 저장 가능

# 8. 결과 화면 시각 강화

## 8.1 목적

결과 화면을 더 뿌듯하고 보기 좋게 만들어 학습 경험을 마무리한다.

## 8.2 사용자 가치

* 끝까지 푼 보람이 커진다.
* 다시 도전하고 싶어진다.
* 결과를 한눈에 이해할 수 있다.

## 8.3 개선 방향

현재 결과 화면에서 아래 항목을 강화할 수 있다.

1. 결과 요약 카드 시각 강화
2. 정답률을 더 크게 표시
3. 최고 연속 정답 표시
4. 칭찬 메시지 강화
5. 별/배지/작은 트로피 같은 시각 요소 추가

## 8.4 추가 표시 항목

* 정답 수
* 오답 수
* 정답률
* 최고 연속 정답 수
* 타이머 모드였다면 시간 초과 수
* 학습 모드(객관식 / 직접 입력)

## 8.5 UI 구성 권장

### 영역 1: 축하 헤더

* 큰 제목
* 작은 반짝이/별 장식
* 요약 칭찬 문구

### 영역 2: 점수 카드

* 총 문제 수
* 정답 수
* 오답 수
* 정답률

### 영역 3: 추가 성과 카드

* 최고 연속 정답
* 시간 초과 수
* 오늘의 학습 모드

### 영역 4: 오답 리스트

### 영역 5: 액션 버튼

* 오답 다시 풀기
* 처음으로
* 필요하면 다시 10문제 풀기

## 8.6 성과 메시지 예시

* 정말 잘했어요!
* 끝까지 풀어서 멋져요!
* 이번엔 최고 연속 4문제를 맞혔어요!
* 다시 도전하면 더 잘할 수 있어요!

## 8.7 구현 포인트

* 카드 분리, 숫자 강조, 뱃지 형태 추가
* 하지만 정보 과다는 금지
* 결과 화면이 예뻐지더라도 오답 다시 풀기 버튼이 묻히면 안 됨

# 9. 학습 기록 저장

## 9.1 목적

사용자의 학습 결과를 저장해 다음에 다시 봐도 이어서 성과를 확인할 수 있도록 한다.

## 9.2 사용자 가치

* 얼마나 했는지 누적해서 볼 수 있다.
* 반복 학습 동기를 만들 수 있다.
* 보호자 입장에서도 학습 흔적을 확인하기 좋다.

## 9.3 권장 범위

초기 저장은 서버 없이 **브라우저 localStorage** 기반으로 구현하는 것을 권장한다.

이유:

* 구현이 빠름
* v0.1 구조를 크게 바꾸지 않음
* 로그인/DB 없이도 가능

## 9.4 저장 단위

한 세션이 끝날 때 아래 정보를 저장한다.

예시:

```ts
type StudyRecord = {
  id: string;
  playedAt: string;
  selectedDan: DanOption;
  answerMode: AnswerMode;
  isTimerMode: boolean;
  totalQuestions: number;
  correctCount: number;
  wrongCount: number;
  accuracy: number;
  bestStreak: number;
  timeoutCount?: number;
};
```

## 9.5 저장 시점

* 결과 화면 진입 시 1회 저장
* 같은 세션이 중복 저장되지 않도록 주의

## 9.6 저장 방식

권장 key:

```ts
'gugu-study-records'
```

저장 구조:

```ts
StudyRecord[]
```

최신 기록이 앞에 오도록 정렬 권장

## 9.7 표시 방식

초기 버전은 아래 중 하나를 선택한다.

### 권장안 A

결과 화면 하단에 최근 기록 3~5개 표시

### 권장안 B

별도 기록 페이지 `/history` 추가

# 12. 수동 테스트 항목

## 직접 입력 모드

* 입력 모드 선택 후 문제 풀이가 가능한지
* 빈 값일 때 채점되지 않는지
* 입력 후 채점되면 다시 수정할 수 없는지

## 타이머 모드

* 문제 시작 시 시간이 줄어드는지
* 답하면 타이머가 멈추는지
* 시간 초과 시 자동 오답 처리되는지
* 다음 문제에서 시간이 초기화되는지

## 연속 정답 보상

* 연속 정답 시 streak가 증가하는지
* 오답 시 streak가 초기화되는지
* 최고 streak가 정확히 기록되는지

## 효과음 on/off

* on 상태에서만 소리가 나는지
* off 상태에서 소리가 완전히 꺼지는지
* 토글 후 즉시 반영되는지

## 결과 화면 강화

* 정답률, streak, 추가 정보가 올바르게 보이는지
* 버튼이 묻히지 않는지

## 학습 기록 저장

* 결과 화면 진입 시 기록이 저장되는지
* 중복 저장되지 않는지
* 새로고침 후에도 최근 기록이 유지되는지

# 13. 완료 기준

다음 조건을 만족하면 v0.2~v0.3 확장 구현이 완료된 것으로 본다.

* 객관식/직접 입력 두 가지 모드를 안정적으로 지원한다.
* 타이머 모드가 상태 꼬임 없이 동작한다.
* 연속 정답 보상이 정확하게 반영된다.
* 효과음 on/off가 일관되게 동작한다.
* 결과 화면이 더 시각적으로 좋아지고 정보 전달력이 높아진다.
* 학습 기록이 localStorage에 안정적으로 저장되고 다시 읽힌다.

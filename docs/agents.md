# AGENTS.md

이 저장소의 AI 에이전트, 코딩 보조 도구, 개발자는 아래 규칙을 기준으로 작업한다.  
대상 프로젝트는 **초등학생용 구구단 연습 웹앱**이며, 목표는 **밝고 예쁜 디자인 + 단순한 학습 흐름 + 안정적인 상태 로직**을 갖춘 MVP를 만드는 것이다.

---

## 1. 프로젝트 목표

이 프로젝트는 초등학생이 구구단을 쉽고 재미있게 연습할 수 있는 웹앱을 만드는 것이 목적이다.

핵심 목표는 아래와 같다.

1. 아이가 혼자서도 쉽게 시작할 수 있어야 한다.
2. 한 화면에서 해야 할 행동이 명확해야 한다.
3. 정답/오답 피드백이 즉시 보여야 한다.
4. 틀린 문제를 다시 풀 수 있어야 한다.
5. 여자 아이가 좋아할 수 있는 밝고 예쁜 분위기를 가져야 한다.
6. 복잡한 기능보다 완성도 높은 기본 흐름을 우선한다.

---

## 2. 제품 범위

초기 버전(v0.1)은 아래 기능만 포함한다.

- 시작 화면
- 단 선택(2단~9단, 전체 섞기)
- 객관식 4지선다 문제 풀이
- 즉시 채점
- 진행률 표시
- 점수 표시
- 결과 화면
- 오답 다시 풀기
- 모바일 대응

초기 버전에서 제외한다.

- 로그인
- 서버 DB 저장
- 보호자 대시보드
- 사용자별 학습 이력 저장
- 음성 기능
- 과한 애니메이션
- 불필요한 캐릭터 시스템
- 백엔드 API 설계

## 3. 기술 스택 원칙

기본 기술 스택은 아래를 따른다.

- Next.js (App Router)
- React
- TypeScript
- Tailwind CSS

추가 원칙:

- 상태 관리는 우선 `useState` 중심으로 한다.
- 전역 상태 라이브러리(Zustand, Redux 등)는 v0.1에서 도입하지 않는다.
- 서버 컴포넌트보다 클라이언트 상호작용 중심 UI를 우선한다.
- 불필요한 외부 라이브러리 추가를 최소화한다.
- 구현은 단순하고 읽기 쉽게 유지한다.

## 4. 디렉토리 구조 원칙

권장 구조는 아래를 따른다.

```txt
src/
  app/
    page.tsx
    globals.css

  components/
    StartScreen.tsx
    QuizScreen.tsx
    ResultScreen.tsx
    QuestionCard.tsx
    ChoiceButton.tsx
    ProgressBar.tsx
    ScoreBoard.tsx

  lib/
    generateQuestions.ts
    generateChoices.ts
    shuffleArray.ts
    calculateAccuracy.ts

  types/
    quiz.ts
````

규칙:

* `app/page.tsx`는 화면 전환과 최상위 상태 관리 담당
* `components/`는 UI 컴포넌트만 담당
* `lib/`는 순수 함수 중심 유틸리티
* `types/`는 공통 타입 정의

## 5. 상태 관리 규칙

핵심 상태는 모두 `app/page.tsx`에 둔다.

예상 상태:

* `screen`
* `selectedDan`
* `questions`
* `currentIndex`
* `score`
* `wrongAnswers`
* `selectedAnswer`
* `isAnswered`
* `feedbackMessage`
* `isRetryMode`

중요 규칙:

1. 상태는 한 곳에서 관리한다.
2. 자식 컴포넌트는 상태 소유자가 아니라 표시자여야 한다.
3. 한 문제는 한 번만 채점한다.
4. 답을 선택한 뒤 다시 점수 계산이 일어나면 안 된다.
5. 다음 문제로 넘어갈 때 현재 문제 상태를 초기화한다.
6. 오답 다시 풀기는 새 세션처럼 처리한다.

금지:

* 여러 컴포넌트가 같은 핵심 상태를 따로 들고 있는 구조
* 점수와 오답 기록을 자식 컴포넌트에서 직접 수정하는 구조
* `isAnswered` 없이 중복 채점 가능한 구조

## 6. 화면 구조 규칙

허용 화면:

### StartScreen

* 제목 표시
* 단 선택
* 시작 버튼

### QuizScreen

* 진행률 표시
* 문제 표시
* 객관식 선택
* 정답/오답 피드백
* 다음 문제 버튼

### ResultScreen

* 결과 요약
* 오답 목록
* 오답 다시 풀기
* 처음으로 가기

## 11. 컴포넌트 작성 규칙

컴포넌트는 단순하고 명확하게 작성한다.

원칙:

* 한 컴포넌트는 한 역할만 담당
* props 이름은 직관적으로 작성
* UI 컴포넌트 내부에 복잡한 비즈니스 로직을 넣지 않는다
* 재사용성보다 이해 쉬운 구조를 우선한다

예:

* `QuestionCard`는 문제 표시만 담당
* `ChoiceButton`은 보기 버튼 렌더링만 담당
* 정답 판정 로직은 부모에서 처리

* 모바일에서도 무리 없이 사용할 수 있다

## 22. 에이전트 작업 지침

AI 에이전트는 아래 원칙을 따라 작업한다.

* 먼저 기존 구조를 이해하고 수정한다.
* 임의로 파일 구조를 크게 바꾸지 않는다.
* 새 파일 추가 시 역할이 명확해야 한다.
* UI를 수정할 때 기능 흐름을 깨지 않는다.
* 로직을 수정할 때 `isAnswered`, `score`, `wrongAnswers`의 일관성을 최우선으로 본다.
* 디자인 수정 시 밝고 예쁜 분위기를 유지하되 가독성을 해치지 않는다.
* 사용자가 명시하지 않은 기능을 마음대로 추가하지 않는다.
* 구현 후에는 최소한의 수동 테스트 시나리오를 기준으로 점검한다.

---

## 23. 수동 테스트 기준

최소 아래 시나리오는 항상 점검한다.

1. 단을 선택하지 않으면 시작 버튼이 비활성화되는지
2. 단 선택 후 시작이 되는지
3. 답을 누르면 즉시 채점되는지
4. 같은 문제에서 두 번 채점되지 않는지
5. 마지막 문제 후 결과 화면으로 가는지
6. 오답이 있으면 다시 풀기 버튼이 보이는지
7. 오답이 없으면 축하 문구가 자연스럽게 보이는지
8. 모바일 폭에서도 버튼이 잘 눌리는지
